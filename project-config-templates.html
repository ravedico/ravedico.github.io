Case study: Configuration templates

Designing a web-based application management tool for data engineers

Company: Spot by NetApp | Ocean for Apache Spark
Role: UX Designer and Researcher
Timeline: 9 months
Deliverables: UX research and insights, wireframes, prototypes, user testing
Teammates: Product Manager, fullstack engineers

Overview

Spot by NetApp is a CloudOps company helping data engineers with the automation and optimization of big-data applications. My role was to improve the experience and process of managing configurations of big data applications.
As the Lead UX Designer for the Big-data product, I handled this project end-to-end including user research and user experience design. Of course, it is a multi-team effort with collaboration between design, engineering, and product teams.

The challenge

Data engineers faced with major challenges in configuring Apache Spark applications:

  • Manual configuration processes requiring deep technical expertise
  • Setup times extending to several hours per application
  • Frequent need for DevOps support
  • High risk of misconfigurations affecting production systems
  • Lack of version control and configuration history

Our primary goal was to empower data engineers to independently create and maintain their application configurations while reducing setup time and configuration errors.

Research Process

Initial Discovery

We began with a solid foundation:

  • Existing user personas from active customers
  • Usage data from current config-template features
  • Internal subject matter experts
Stakeholder Interviews

We conducted interviews with internal stakeholders focusing on:

  • Current usage patterns of config-templates
  • Desired capabilities for the revamped system
  • User interactions and workflows
  • Short-term and long-term user objectives
  • Current pain points and obstacles
Surveys

We composed a series of open-ended and closed questions to rate features, and to get qualitative and quantitative insights which were sent to the top active customers. We asked to rank ideas and it helped us to validate our assumptions

Customer Interviews

We conducted five interview sessions with major usage customers. We asked about their daily workflow involving creating and maintaining config-templates, integration with existing tools and processes and other aspects of their daily work activity within our product and outside it.

Solutions ROI

Key Research Findings

  1. Users strongly desired version control and change history:
    "I need an option to view what changes were made in the templates over time and the reasons they were made"
    - User interview participant
  2. Template organization was a major pain point: Need for logical grouping and categorization and no way to track template purposes, and uses per application
  3. Search and filtering capabilities were crucial for large template libraries

Design Solution

Feature Priority List

Based on research findings, we prioritized:

  1. Design a page with version history and changelog:
    • Stating who made the changes and why.
    • An option to revert to a previous version.
    • Selecting 2 versions for comparison.
  2. Tagging system - This feature did not exist in the Spot console, and deciding on its behaviour was challenging:
    • Creating a new tag.
    • Adding and removing a tag from a template.
    • Tag structure - key:value.
    • Tags management system.
    • Advanced filtering by tags mechanism.
  3. Advanced search and filtering.
  4. Template comparison tools
  5. Applications running on config-templates list
  6. Presets for templates with common configurations:
    • A screen with a list of the presets dedicated to creating new config-templates based on pre-created presets.
    • Contains starter presets and user-created presets.
    • An option to make a preset from existing config-template.
  7. Bulk actions for template management:
    • Add and remove tags.
    • Compare two templates.
    • Delete templates.
Design Process
  1. Analyzed existing design system components
  2. Researched applicable design patterns
  3. Created innovative solutions for unique challenges:
    • Bulk action interfaces
    • Version comparison views
    • The relations of templates and applications
Major challenges

Main page - config-templates list:
The visual features we added, such as filtering and tagging, increased the data for each item. After several tests, it was decided to change the layout from a cards view with a wide template code editor space to a table view, which easily facilitates more data and bulk actions. Since this solution came at the expense of the code editor, we added an option of viewing it in a full-screen view.

Tagging system
We introduce tagging options to the console. It's a major feature that can make a significant impact on all the products and pages. We had to make prioritisation decisions build a simple mechanism at first, and collaborate with other teams and management to create a whole tags management system in the next roadmap.

Success measurement

We will know our success rate when:

  • We can monitor a reduced troubleshooting and config-template creation time
  • Users will report decreased DevOps support requests
  • More new users per team will start working with the config-template
  • Reduced amount of unused config-template per customer
